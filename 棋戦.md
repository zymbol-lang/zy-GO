# 棋戦 — the complexity test

`棋戦.zy` plays the AI against itself, over and over, and measures what happens.
It exists to answer four questions that no unit test can:

1. **Does the interpreter slow down as a position fills up, and by how much?**
2. **Does memory grow across hundreds of games?** — the question v0.0.8 exists
   to answer, since automatic destruction at last use is its headline change.
3. **Do the difficulty levels mean anything?** A level that does not beat a
   weaker one is a label, not a level.
4. **Does personality change outcomes, or only style?**

It also produces the thing all of that is for: a directory of replayable games.

```bash
zymbol run 棋戦.zy [games] [board] [mode]

zymbol run 棋戦.zy                    # 10 games, 9×9, drawn continuously
zymbol run 棋戦.zy 200 9 静           # 200 games, no drawing, one line each
zymbol run 棋戦.zy 1 19 pausa         # one 19×19 game, step by step
```

| Argument | Values | Default |
|----------|--------|---------|
| games | any count | 10 |
| board | 9 · 13 · 19 | 9 |
| mode | `pausa` · `continua` · `静` | `continua` |
| lang | `en` · `es` | `en` |

Mode and language are recognised by what they say, not by where they sit, so
`1 19 es` and `200 9 静 en` both do what they look like they do.

English is the default because the records are meant to be read by a program
later, and because the interesting thing about a benchmark is its numbers, not
its prose. **The records in `棋譜/` are never translated** — they keep one
stable machine form whatever the display language is, since a dataset split
across two languages is two datasets.

| Mode | Behaviour |
|------|-----------|
| `pausa` | Draws every move and waits for `↵`. Study mode — watch a position develop one stone at a time. |
| `continua` | Draws every move and keeps going. The default. The first moves flash past; the last ones visibly crawl. |
| `静` (or `silencio`) | Draws nothing, prints one line per game. Batch mode, for the runs that take an hour. |

## Why this is not a TUI

`対局.zy` — and Serpiente, and Hov veS — open a `>>| { }` block: alternate
screen, raw mode, cleaned up on exit. **棋戦 does not**, and the difference is
deliberate.

`>>|` refuses to run when output is not a terminal:

```bash
$ zymbol run harness.zy > run.log
Runtime error: failed to enable raw mode: No such device or address
```

A benchmark has to survive being redirected. `zymbol run 棋戦.zy 200 9 静 > run.log`
left alone for an hour is the normal way to use this, and a harness that only
works with someone watching is not a harness.

The alternate screen is the second reason. Everything drawn inside `>>|`
disappears when the block ends — which is exactly right for a game, and exactly
wrong for a run whose final report you want left on screen and in your
scrollback.

Nothing is given up for it. `>>~` positions output perfectly well on its own;
only `<<|` (single keypress) needs raw mode, and `pausa` mode waits on `↵`
through ordinary line input instead. The board is drawn with `>>!` and `>>~`
into the normal screen, the report is printed after it, and Ctrl-C behaves the
way Ctrl-C is supposed to.

So: a game is a TUI, an instrument is a program that prints. `集計.zy` never
draws at all, for the same reason.

## Measuring time without a clock

Zymbol has no time primitive. `@~` sleeps; nothing reads the clock. So the
timestamps come from the shell:

```zymbol
時刻() {
    t = <\ "date +%s%N" \>
    <~ #|t|
}
```

That call is not free — it forks a shell. Measured on the machine that wrote
this, **one round trip costs about 2.9 ms**, which is a large fraction of a fast
move and would quietly inflate every early measurement.

So the harness calibrates at startup: it takes eight timestamps back to back,
averages the intervals, and subtracts that from every measurement afterwards.
The figure is printed in the run header, because a measurement whose error bar
is hidden is worse than no measurement:

```
  clock cost   2871 µs per timestamp (subtracted)
```

Two consequences worth stating plainly:

- **Timings below about 5 ms are noise.** The correction is an average, and the
  variance of a fork is not small.
- The measurement is of *the engine's thinking*, not of drawing or of I/O. The
  interval brackets exactly one call to `思考::選択`.

Memory is read the same way, once per game rather than once per move:

```zymbol
記憶() {
    m = <\ "ps -o rss= -p $PPID | tr -d ' '" \>
    <~ #|m|
}
```

`$PPID` inside the spawned shell is the `zymbol` process itself — the resident
set of the interpreter running the games.

## Reproducibility is the point

`std/random` keeps its state hidden. That is fine for a game and useless for a
study: a recorded game that cannot be replayed records *what* happened without
recording *why*.

So 棋戦 uses an explicit LCG whose seed the caller threads through — the same
approach [Serpiente](https://github.com/zymbol-lang/zySerpiente) uses, for the
same reason:

```zymbol
乱数(種) { <~ (1664525 * 種 + 1013904223) % 2147483647 }
```

`思考::選択(…, 種<~)` advances the seed through an output parameter. Every game
gets its own seed derived from a master seed, and **both are recorded**: the
master in the run header, the game's own in its file. Give the same seed and the
same game comes back, move for move.

The master seed itself comes from three independent entropy sources, the way
Serpiente does it:

```zymbol
_ns  = <\ "date +%N" \>
_pid = <\ "echo $$" \>
_rnd = <\ "od -An -N2 -tu2 /dev/urandom | tr -d ' \n'" \>
親種 = (#|_ns| + #|_pid| * 31337 + #|_rnd| * 65537) % 2147483647
```

## The records — games/

**One directory per run**, holding one file per game plus the run's own summary:

```
games/
  20260722_003241_GO9x9/
    0001.kifu
    0002.kifu
    …
    0200.kifu
    summary.txt
  20260722_091510_GO19x19/
    0001.kifu
    …
```

The unit that matters is the run, not the file. A run that goes wrong — a bad
weight, a crash halfway, a parameter you meant to change — is one `rm -rf` away,
and every other run survives it untouched. Nothing is ever overwritten: two runs
a second apart get different directories.

Set `ZYGO_KIFU` to write straight into a clone of the dataset repository:

```bash
ZYGO_KIFU=~/src/zy-GO-kifu zymbol run --vm 棋戦.zy 200 9 静
```

```

white level=3 personality=2
```

Deliberately line-based and greppable rather than JSON: every line is one fact,
`grep`, `cut` and `awk` all work on it, and a future reader in any language
needs no parser. Each move carries its own timing, so a run of 200 games is
also a dataset of ~20,000 timed decisions.

This is the format a learning system would eat later. It has the position's
history, the choices made, the levels and personalities that made them, and the
seed that reproduces the lot.

### summary.txt and stats.txt

Two files, because they answer different questions.

**`summary.txt` is what a person reads** — the report exactly as it scrolled
past, in whatever language the run was made in, including the one line per game
that batch mode prints:

```
     1  ⚫Advanced     守 vs ⚪Advanced     守   W+1.5   400 moves   31018 ms
     2  ⚫Intermediate 守 vs ⚪Intermediate 攻   W+14.5  400 moves   38664 ms
     …

── time per move ──
  first 10 moves, avg       38 ms
  last 10 moves, avg        128 ms
  growth first→last         ×3.3
```

Those per-game lines are recorded in **every** mode, not only in `静`. In the
drawing modes the screen is busy with a board, so they are written and not
printed — the run's own copy of the report is complete either way.

**`stats.txt` is what a program reads** — always the same keys in the same order
in English, whatever the display language, so runs made months and languages
apart can still be summed:

```

```

Two details worth stating. `time_below_clock_resolution` is the warning as a
flag, so a script aggregating fifty runs can drop the ones whose timings are
noise instead of averaging them in. And the summary is **always** in this
machine form whatever the display language is set to — a summary that changes
shape with a flag cannot be aggregated across runs.

The matrices are written raw as wins and games rather than as percentages, so
runs can be summed before any division. Percentages of percentages are a
classic way to get a wrong answer confidently.

## Personality

Each side gets a level **and** a personality, drawn independently:

| | Effect |
|---|---|
| 均衡 (balanced) | weights as designed |
| 攻 (offensive) | capture and atari ×150%, escape and connection ×70% |
| 守 (defensive) | escape, connection and self-atari avoidance ×150%, capture and atari ×70% |

The evaluation collects its layers into an attacking bucket and a defending one,
then weighs them. The layers themselves know nothing about personality — which
is what keeps this from becoming a second copy of the evaluator.

Note what personality is *not*: it is meant not to be difficulty. Neither
setting is supposed to play worse; they disagree about which of two good moves
is better.

The measurement, across four hundred games:

```
                   balanced     offensive     defensive
  balanced      50% /86       63% /111      50% /80
  offensive     36% /111      50% /88       59% /82
  defensive     50% /80       40% /82       50% /80
```

Balanced beats offensive 63% of 111 games. Offensive beats defensive 59% of 82.
And balanced against defensive is exactly even across 80.

That is **not a strength ordering** — if it were, balanced would beat defensive
too. It is closer to rock-paper-scissors with one edge missing, and at these
sample sizes the two live edges are unlikely to be chance while the shape they
form is not yet explained. It needs several hundred more games before it earns
a mechanism.

**Two earlier readings of this same question were wrong, and both are recorded
here on purpose.** At 30 games the answer was "offensive > balanced > defensive,
consistent in every cell", and this document reasoned at length about why that
would follow from area scoring. At 200 games the answer was "all noise, every
cell between 41% and 58%", and the 57% balanced-over-offensive was dismissed as
nothing — it was the signal, and it grew. Each time the data was four to a
hundred games per cell and each time the conclusion outran it.

The habit that produced both errors is the same one: a plausible mechanism and
a small sample will agree with each other whenever you want them to. Hence
`集計.zy`, hence counts rather than percentages in `summary.txt`, and hence the
"cells under 30 games say nothing" printed under every matrix.

## Levels

The level is a single number: **how far below the best a move may score and
still be picked** — 60, 25 or 5 points.

That is a deliberate model of what a weak player is. A beginner is not someone
who plays badly on purpose; it is someone who cannot tell a good move from a
nearly-good one. Widening the margin makes the engine less certain, not more
self-destructive: it still never fills its own eyes and still takes a free
capture, because those are decided before the margin is applied.

The report's win matrix is the test of whether that model works. If 上級 does
not beat 初級 across enough games, the levels are decoration.

**Partly.** Four hundred games on 9 × 9 across two runs, aggregated by
`集計.zy`, levels drawn at random per side:

```
                   Beginner    Intermediate    Advanced
  Beginner      50% /90       10% /109      5% /88
  Intermediate  89% /109      50% /60       48% /94
  Advanced      94% /88       51% /94       50% /68
```

At the bottom the ladder is emphatic: Beginner takes 10% against Intermediate
and 5% against Advanced. Every same-level cell sits at 50%, which is what a fair
matrix must show and a useful check that the tally is not counting something
twice.

**Between Intermediate and Advanced there is nothing** — 48% across 94 games.
The only trace of a difference is indirect: Beginner does worse against Advanced
(5%) than against Intermediate (10%), so something separates them, but not
enough to survive a head-to-head.

Widening the acceptance margin from 5 points to 25 changes almost nothing;
widening it from 25 to 60 changes everything.

That is a finding about the model, not about the engine. "How far below the best
a move may score" captures what separates a beginner from a competent player and
stops capturing anything after that — because at a 25-point margin the engine is
already picking the best move nearly always, and tightening to 5 only removes
choices it was rarely making. A genuinely stronger 上級 needs another *layer*,
not a narrower margin. Ladder reading is the obvious candidate.

An earlier 30-game run had Advanced beating Beginner 12 times out of 12 and was
reported here as settling the question. It did settle that one; it also showed
Intermediate against Advanced even at four games, which was dismissed as too
small a sample. Two hundred games say the even result was the real one.

## Reading the report

```
── time per move ──
  first 10 moves, avg   79 ms
  last 10 moves, avg    161 ms
  slowest final stretch 292 ms per move
  whole run             580 s of thinking
  growth first→last     ×2.0
```

The growth ratio is computed in tenths on purpose — integer division would round
a 2.0× growth down to 2× and a 1.7× one down to 1×, hiding the very thing the
test exists to show.

**Why it grows.** The candidate set widens as stones appear: on an empty board
the opening book confines the choice to a handful of corner points, while at
move 80 nearly every empty intersection is within reach of a stone and must be
scored. Each candidate is scored on a *copy* of the board, and each score walks
chains and liberties. The cost is roughly (number of candidates) × (cost of a
chain walk), and both terms rise together.

```
── memory (RSS) ──
  before first game  9432 KB
  after last game    12848 KB
  growth             3416 KB over 30 games
  peak               12848 KB
```

This is the v0.0.8 number. Automatic destruction at last use is that release's
headline change, and the question it has to answer is whether a program that
allocates a board per legality test — hundreds of thousands of arrays over a run
— settles at a plateau or climbs without bound.

**It plateaus.** Runs of different lengths, same machine:

| Games | Engine | Before | After | Growth |
|-------|--------|--------|-------|--------|
| 3 | tree-walker | 9660 KB | 12044 KB | +2384 KB |
| 4 | tree-walker | 9612 KB | 13236 KB | +3624 KB |
| 6 | tree-walker | 9600 KB | 12528 KB | +2928 KB |
| 30 | tree-walker | 9432 KB | 12848 KB | +3416 KB |
| **200** | **`--vm`** | **8396 KB** | **8492 KB** | **+96 KB** |

Under the tree-walker, ten times the games produce the same final figure: the
resident set climbs to a working set around 12–13 MB in the first game or two
and stays there. Under the VM, two hundred games — roughly twenty thousand
moves and several million board allocations — moved it by **96 KB**.

That is the answer auto-free was supposed to give, and the VM's is close to
flat. The two engines are not directly comparable (different allocation
strategies, different baselines), but neither shows growth with the number of
games, which is the property that matters.

Peak and final are reported separately because a peak above the final figure is
direct evidence of reclamation. In the 6-game run the peak was 12612 KB against
a final 12528 KB; in the 30-game run they coincide.

## Aggregating runs — 集計.zy

A single run is a sample. The numbers that matter only appear once runs are
summed:

```bash
zymbol run 集計.zy                    # everything under games/
zymbol run 集計.zy ~/src/zy-GO-kifu   # a whole dataset
zymbol run 集計.zy games es           # in Spanish
```

```
═══ 集計 — every run together ═══
  directory                 /path/to/zy-GO-kifu
  runs                      2
  games                     400
  unfinished (no summary)   1
```

It reads the **records**, not the summaries. A run interrupted before it wrote
its summary still has its games and they still count — the harness only writes
`summary.txt` at the end, so its absence is exactly the marker for "this run did
not finish", reported rather than hidden.

Sums are over counts, never over percentages: `summary.txt` stores wins and
games raw for the same reason. Averaging percentages across runs of different
sizes is a classic way to be confidently wrong.

Two things it taught while being written, both about `1..n`:

- **`@ i:1..n` counts *down* when `n` is 0.** Descending ranges are a
  documented feature — `5..1` walks 5,4,3,2,1 — so `1..0` walks `[1, 0]`, and a
  count loop over a possibly-empty array indexes element 1 of it on the way
  past. Every such loop needs a guard.
- `Char` is still not comparable, so `c >= '0'` is a runtime error. Digits are
  recognised by code point through `表示/文字::符号点`, the same route the width
  table takes.

## Tree-walker versus VM

The harness runs under both engines, and because a game replays from its seed,
the same game can be run in each and compared exactly. It is:

| Workload | Tree-walker | `--vm` | Ratio |
|----------|-------------|--------|-------|
| 30 moves on 19 × 19, isolated | 17.8 s | 1.29 s | **13.9×** |
| One full 9 × 9 game | 21.1 s | 2.19 s | **9.6×** |
| One full 19 × 19 game | 6 min 40 s | 49.8 s | **8.0×** |

Both engines produced identical games — same seed, same moves, same result — so
this is not the VM skipping work.

**This is far more than the ~4.4× the project's own documentation claims.** That
figure comes from arithmetic-heavy benchmarks; this workload is dominated by
array allocation, copying and traversal — a board copy per legality test, a
visited array per flood fill, hundreds of thousands of them per game — and the
register VM turns out to be much better at that than the ~4× number suggests.
`interpreter/ARCHITECTURE.md` understates its own VM.

Note what this does **not** mean for the game: `対局.zy` runs the tree-walker
because a human move takes seconds of human thought, so 180 ms versus 20 ms is
invisible. The VM matters here, where the machine plays both sides for an hour.

### The instrument has a floor

Under `--vm` on 9 × 9 a move costs about 5 ms, and a timestamp costs about 3 ms.
The measurement has run out of resolution, and the growth ratio comes out
**below 1** — which is impossible, and is the number saying so.

The harness detects this and says it plainly rather than reporting nonsense with
a straight face:

```
  growth first→last         ×0.7

  ⚠ below the clock's resolution — these timings are noise
  use a bigger board, or the tree-walker, to measure this
```

The rule of thumb it applies: if the average early move costs less than three
timestamps, the timings are not measuring the engine, they are measuring
`date`. For VM timings, use 19 × 19.

## What this is for

Three things, in order of how soon they pay off:

1. **A regression net for performance.** A change to the evaluator that doubles
   the cost per move is invisible in a unit test and obvious here.
2. **Evidence about the memory model.** v0.0.8's central claim, measured over a
   workload that allocates hard for an hour.
3. **A dataset.** Twenty thousand timed, reproducible decisions with the level
   and personality that produced each one. When a native tensor type makes a
   learned engine viable — see `ROADMAP_IA.md` in
   [Zofía](https://github.com/zymbol-lang/zy-Zofia) — this is what would feed
   it, and generating it is the part that has to exist first.

Everything here is Zymbol, including the clock and the statistics.
